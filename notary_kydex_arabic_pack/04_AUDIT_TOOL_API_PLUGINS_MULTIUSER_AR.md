# أداة تدقيق إضافات وواجهات API متعددة المستخدمين والتطبيقات


> حزمة عمل عربية مبنية على خلاصة نقاشات بيئة كتاب العدل وKYDEX، وتشمل: محرك التحقق، موقع مجلس كتاب العدل، صفحة/ثيم الكاتب العدل، إضافات الربط API، تحليل عرض JAB، والطرح المقابل القائم على خدمات Web/SaaS بدلاً من تطبيقات الهاتف.

> ملاحظة منهجية: هذه الوثائق هي أدوات تدقيق واستراتيجية ومواصفات أولية. لا تُعد رأياً قانونياً نهائياً، ويجب مراجعتها من مستشار قانوني مختص قبل اعتمادها تعاقدياً أو تنظيمياً.



## منهجية التقييم الموحدة

يعتمد التدقيق على سلم من 0 إلى 5:

| الدرجة | المعنى |
|---|---|
| 0 | غير موجود إطلاقاً |
| 1 | مذكور نظرياً فقط |
| 2 | موجود جزئياً دون اختبار أو توثيق |
| 3 | موجود ويعمل أساسياً |
| 4 | موجود ومختبر ومتكامل مع بقية النظام |
| 5 | موجود، مختبر، موثق، قابل للتدقيق، ومناسب للإطلاق |

قرار التدقيق:

- أقل من 2.5: غير قابل للاعتماد.
- من 2.5 إلى 3.5: يحتاج تحسينات جوهرية.
- من 3.5 إلى 4.5: قابل للتجربة المحدودة.
- من 4.5 إلى 5: قابل للإطلاق المنضبط.


## 1. الفكرة المركزية

المطلوب ليس API واحداً بسيطاً، بل طبقة ربط تسمح لعدة مستخدمين وعدة تطبيقات بالاستعلام من KYDEX، بينما يتولى KYDEX الوصول إلى عدة مصادر بيانات. أي أن العلاقة هي many-to-many:

مستخدمون متعددون ← تطبيقات متعددة ← KYDEX API ← مصادر متعددة.

## 2. محاور التدقيق

### 2.1 إدارة العملاء والمفاتيح

| البند | الدليل المطلوب | الدرجة |
|---|---|---|
| حساب لكل عميل/مؤسسة | tenant model | 0-5 |
| API keys لكل تطبيق | key table + scopes | 0-5 |
| Hashing للمفاتيح | لا تخزين مكشوف | 0-5 |
| إيقاف/تدوير المفاتيح | revoke/rotate endpoints | 0-5 |
| تحديد الصلاحيات scopes | screen, bulk, monitor | 0-5 |

### 2.2 نقاط النهاية Endpoints

| Endpoint | الوظيفة | معيار القبول |
|---|---|---|
| POST /screen | فحص اسم واحد | نتيجة + score + sources |
| POST /bulk-screen | فحص دفعة | job id + progress |
| GET /bulk-screen/:id | حالة الدفعة | completed/failed/result |
| POST /monitor | مراقبة مستمرة | monitor id |
| GET /sources | المصادر المتاحة | list + status |
| GET /usage | الاستخدام | counts + billing data |

## 3. نموذج المسؤولية

- صاحب API key مسؤول عن الاستخدام المصرّح به.
- KYDEX مسؤول عن التشغيل، التحديث، وحفظ سجلات الاستعلام.
- مصادر البيانات تبقى محفوظة ومحدثة داخل KYDEX ولا تعطى مباشرة لكل تطبيق.
- كل استعلام يجب أن يحمل tenant_id وapi_key_id وtimestamp وsource_versions.

## 4. مؤشرات الخطر الأحمر

- API key داخل الواجهة الأمامية.
- غياب rate limiting.
- عدم وجود سجلات استخدام.
- عدم القدرة على تعطيل مفتاح مخترق.
- استعلام مباشر من plugin إلى OFAC/UN بدل المرور عبر KYDEX.

## 5. قرار الجاهزية

لا يعتبر API جاهزاً للبيع SaaS قبل اكتمال: المفاتيح، الصلاحيات، العزل، السجلات، الحدود، وتوثيق المطورين.
