# العرض العربي المقابل — خدمات Web/SaaS بدلاً من تطبيق الهاتف


> حزمة عمل عربية مبنية على خلاصة نقاشات بيئة كتاب العدل وKYDEX، وتشمل: محرك التحقق، موقع مجلس كتاب العدل، صفحة/ثيم الكاتب العدل، إضافات الربط API، تحليل عرض JAB، والطرح المقابل القائم على خدمات Web/SaaS بدلاً من تطبيقات الهاتف.

> ملاحظة منهجية: هذه الوثائق هي أدوات تدقيق واستراتيجية ومواصفات أولية. لا تُعد رأياً قانونياً نهائياً، ويجب مراجعتها من مستشار قانوني مختص قبل اعتمادها تعاقدياً أو تنظيمياً.

## 2. الملخص التنفيذي

يقترح هذا العرض اعتماد مقاربة Web/SaaS بدلاً من بناء تطبيقات iOS وAndroid. المقاربة الجديدة تحافظ على جميع الوظائف التي وردت في عرض JAB، لكنها تنقلها إلى بيئة أكثر ملاءمة لطبيعة عمل الكاتب العدل: صفحة المجلس، صفحة لكل كاتب عدل، بوابة دخول مكتبية، ومحرك KYDEX للفحص عبر API.

هذه المقاربة تخفض الكلفة الأولية، تقلل التعقيد، تسرّع الإطلاق، وتحوّل المشروع من تطبيق محدود إلى خدمة تشغيل مستمرة.

## 3. لماذا لا نحتاج iOS وAndroid؟

- العمل الفعلي للكاتب العدل يتم في المكتب وليس أثناء التنقل.
- الملفات والسندات والطباعة والتوثيق تحتاج شاشة ومكتباً لا هاتفاً فقط.
- Web Platform يعمل على الحاسوب والهاتف والتابلت دون متجر تطبيقات.
- أي تحديث في الويب يصل فوراً لكل المستخدمين دون تحميل نسخة جديدة.
- الصيانة والأمن والتحكم بالبيانات أسهل بكثير.

## 4. الوظائف المطلوبة كما في عرض JAB، لكن بصيغة Web/SaaS

| وظيفة JAB | التنفيذ المقابل المقترح |
|---|---|
| Android Application | Web/PWA responsive يعمل على الهاتف والحاسوب |
| iOS Application | نفس منصة Web دون تطبيق منفصل |
| API | KYDEX API مستقل وقابل للربط بعدة أنظمة |
| Web Admin | لوحة إدارة موسعة للمجلس والتشغيل |
| دليل كتاب العدل | صفحة عامة + بحث + خريطة |
| أقرب كاتب عدل | خريطة Web مع فلترة منطقة |
| حساب لكل كاتب عدل | بوابة دخول آمنة مع أدوار وصلاحيات |
| بحث في القوائم السوداء | KYDEX Screening Engine |
| Soundex | Soundex + Fuzzy + Arabic normalization |
| عرض المصدر والتاريخ | مصدر + نسخة قائمة + تاريخ تحديث + درجة تطابق |
| تعديل الهاتف والعنوان | Profile update مع موافقة المجلس عند الحاجة |
| تعليق مستخدم | User suspension + reason + audit log |
| إشعارات | In-app/email notifications |
| Session timeout | سياسة جلسات وأمان |
| active/inactive users | لوحة مراقبة نشاط المستخدمين |

## 5. مكونات الحل

### 5.1 صفحة مجلس كتاب العدل

تتضمن:

- الصفحة الرئيسية.
- التعريف بالمجلس.
- التعاميم والأخبار.
- دليل كتاب العدل.
- خريطة تفاعلية.
- صفحة أسئلة شائعة.
- بوابة دخول الأعضاء.
- خيار الربط بمحرك KYDEX.

### 5.2 صفحات الكتاب العدل الفردية

كل كاتب عدل يمكن أن يحصل على صفحة/موقع مصغر يتضمن:

- الاسم والصفة.
- العنوان والخريطة.
- الهاتف والبريد.
- ساعات العمل.
- نموذج طلب موعد.
- روابط المجلس.
- ربط اختياري بخدمة الفحص عند الترخيص.

### 5.3 محرك KYDEX للفحص

الوظائف:

- فحص اسم واحد.
- فحص جماعي bulk.
- مصادر متعددة: OFAC، UN، قوائم محلية/يدوية.
- تطبيع الأسماء.
- مطابقة تقريبية وصوتية.
- تصنيف مخاطر.
- سجل تدقيق.
- ملف أدلة عند الخطر العالي.

### 5.4 لوحة إدارة المجلس

تتضمن:

- إدارة المستخدمين.
- تفعيل/تعطيل الحسابات.
- إدارة المحتوى والتعاميم.
- متابعة الاستخدام.
- مراقبة عمليات الفحص.
- إدارة مصادر البيانات.
- مراجعة الحالات العالية المخاطر.

### 5.5 API plugins والربط الخارجي

يمكن ربط KYDEX بـ:

- موقع المجلس.
- صفحات WordPress للكتاب العدل.
- تطبيقات خارجية مرخصة.
- بوابات مؤسساتية.
- أنظمة داخلية.

## 6. نموذج التشغيل المستمر

### 6.1 التطوير

بناء المنصة والواجهات والربط.

### 6.2 التدريب

تدريب المجلس والكتاب العدل على الاستخدام والتفسير والتصعيد.

### 6.3 الصيانة

إصلاح الأخطاء، تحديثات أمنية، مراقبة الأداء، نسخ احتياطي.

### 6.4 تحديث البيانات اليومي

تحديث قوائم العقوبات، حفظ النسخ، مراقبة الفشل، تنقية البيانات، وتدقيق التحديثات.

### 6.5 إدارة التشغيل

إدارة المستخدمين، التقارير، مراجعة الحالات، دعم الاستفسارات.

## 7. نموذج السعر المقترح

### 7.1 تأسيس أولي منخفض

يمكن تقديم مرحلة تأسيسية محدودة ضمن سقف منخفض إذا اقتصر النطاق على:

- صفحة المجلس الأساسية.
- قالب كاتب عدل فردي.
- واجهة بحث أولية.
- حسابات أولية.
- ربط API أساسي.

### 7.2 اشتراك شهري/سنوي

الاشتراك يغطي:

- الاستضافة.
- تحديث القوائم.
- الصيانة.
- الدعم.
- النسخ الاحتياطي.
- مراقبة الأمن.
- تحسين المطابقة.

## 8. الرسالة التجارية للمجلس

إذا كان لدى المجلس عرض مجاني كامل لبناء التطبيق والمحرك والتشغيل والتحديث والصيانة، فمن الطبيعي اعتماد العرض المجاني بشرط توضيح من يتحمل مسؤولية التشغيل والتحديث والأمن. أما إذا كان الخيار بين كلفة تطبيق هاتف وكلفة خدمة Web/SaaS مستمرة، فالخيار الثاني أكثر ملاءمة واستدامة.

## 9. شروط النجاح

- عدم الاكتفاء بتطبيق هاتف.
- عدم إطلاق البحث دون audit trail.
- عدم الفحص دون حفظ نسخة القائمة.
- عدم السماح بحالات HIGH/CRITICAL دون تصعيد.
- عدم ربط أي تطبيق خارجي دون API key وصلاحيات.
- عدم إهمال التدريب والصيانة اليومية.

## 10. الخلاصة

المقاربة المقترحة تقدم كل وظائف عرض JAB، لكنها تضيف إليها البنية التي تجعلها قابلة للاستخدام المهني: Web، API، دليل، صفحات فردية، امتثال، أدلة، تصعيد، تدريب، صيانة، وتشغيل يومي.

الفرق الأساسي:

> JAB يطرح تطبيقاً. أما العرض المقابل فيطرح خدمة رقمية متكاملة.
