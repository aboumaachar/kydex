# أداة تدقيق محرك KYDEX للفحص والتحقق


> حزمة عمل عربية مبنية على خلاصة نقاشات بيئة كتاب العدل وKYDEX، وتشمل: محرك التحقق، موقع مجلس كتاب العدل، صفحة/ثيم الكاتب العدل، إضافات الربط API، تحليل عرض JAB، والطرح المقابل القائم على خدمات Web/SaaS بدلاً من تطبيقات الهاتف.

> ملاحظة منهجية: هذه الوثائق هي أدوات تدقيق واستراتيجية ومواصفات أولية. لا تُعد رأياً قانونياً نهائياً، ويجب مراجعتها من مستشار قانوني مختص قبل اعتمادها تعاقدياً أو تنظيمياً.



## منهجية التقييم الموحدة

يعتمد التدقيق على سلم من 0 إلى 5:

| الدرجة | المعنى |
|---|---|
| 0 | غير موجود إطلاقاً |
| 1 | مذكور نظرياً فقط |
| 2 | موجود جزئياً دون اختبار أو توثيق |
| 3 | موجود ويعمل أساسياً |
| 4 | موجود ومختبر ومتكامل مع بقية النظام |
| 5 | موجود، مختبر، موثق، قابل للتدقيق، ومناسب للإطلاق |

قرار التدقيق:

- أقل من 2.5: غير قابل للاعتماد.
- من 2.5 إلى 3.5: يحتاج تحسينات جوهرية.
- من 3.5 إلى 4.5: قابل للتجربة المحدودة.
- من 4.5 إلى 5: قابل للإطلاق المنضبط.


## 1. نطاق التدقيق

تُستخدم هذه الأداة لتقييم جاهزية KYDEX بوصفه محرك فحص وتحقيق امتثال، وليس مجرد خانة بحث. يجب أن يتحقق المدقق من أن النظام قادر على استقبال طلب فحص، تطبيع البيانات، فحصها مقابل مصادر متعددة، عرض نتائج مفهومة، وحفظ دليل تدقيق كامل.

## 2. محاور التدقيق

### 2.1 مصادر البيانات والربط

| بند التدقيق | دليل الإثبات المطلوب | الدرجة |
|---|---|---|
| وجود Connector مستقل لكل مصدر بيانات | كود/وثيقة لكل OFAC/UN/محلي | 0-5 |
| عدم الاعتماد على استعلام live فقط وقت الفحص | قاعدة محلية محدثة ونسخ قوائم | 0-5 |
| حفظ نسخة القائمة المستخدمة في كل فحص | list_version في سجل الفحص | 0-5 |
| وجود آلية إنذار عند فشل التحديث | logs + alerting | 0-5 |
| دعم رفع قوائم يدوية CSV/JSON | upload + validation report | 0-5 |

### 2.2 محرك المطابقة

| بند التدقيق | دليل الإثبات المطلوب | الدرجة |
|---|---|---|
| Exact match | أمثلة اختبار | 0-5 |
| Fuzzy matching | نتائج وعتبات واضحة | 0-5 |
| Soundex أو بديل صوتي | مستند خوارزمية | 0-5 |
| تطبيع الأسماء العربية واللاتينية | transliteration rules | 0-5 |
| Alias matching | حقول aliases وربطها | 0-5 |
| شرح سبب النتيجة | match_reason في API/UI | 0-5 |

### 2.3 تصنيف المخاطر

| مستوى الخطر | معيار مقترح | الإجراء |
|---|---|---|
| LOW | لا تطابق أو تشابه ضعيف | متابعة + تسجيل |
| MEDIUM | تشابه جزئي | ملاحظة عناية واجبة |
| HIGH | تطابق قوي أو مؤشر هوية داعم | إنشاء حالة مراجعة |
| CRITICAL | تطابق شبه مؤكد | تجميد الإجراء وتحضير ملف تصعيد |

### 2.4 سجل التدقيق والأدلة

| بند التدقيق | دليل الإثبات المطلوب | الدرجة |
|---|---|---|
| تسجيل كل فحص | query log | 0-5 |
| تسجيل المستخدم والوقت والـ IP | audit metadata | 0-5 |
| تسجيل نسخة القوائم | source version | 0-5 |
| منع تعديل سجلات التدقيق | immutable / hash-chain | 0-5 |
| توليد evidence package | PDF/JSON package | 0-5 |
| تصدير timeline للحالة | case timeline export | 0-5 |

### 2.5 الأمن والخصوصية

| بند التدقيق | دليل الإثبات المطلوب | الدرجة |
|---|---|---|
| HTTPS/TLS | إعدادات الخادم | 0-5 |
| تشفير كلمات المرور | bcrypt/argon2 | 0-5 |
| API keys hashed | schema + tests | 0-5 |
| RBAC | Route protection matrix | 0-5 |
| 2FA للمسؤولين | شاشة/اختبار | 0-5 |
| عزل المستأجرين/العملاء | tenant_id enforcement | 0-5 |
| نسخ احتياطي مشفر | backup logs | 0-5 |

## 3. أسئلة التدقيق الحاسمة

1. هل يمكن إثبات أن الفحص تم على نسخة معينة من OFAC/UN؟
2. هل يستطيع النظام شرح سبب كل match؟
3. هل يعالج اختلافات كتابة الأسماء العربية والإنكليزية؟
4. هل يمنع المستخدم العادي من الوصول إلى ملفات الأدلة الحساسة؟
5. هل يوجد مسار مراجعة عند HIGH/CRITICAL؟
6. هل يمكن للمدقق إعادة بناء القرار بعد سنة أو خمس سنوات؟

## 4. مؤشرات الخطر الأحمر

- نتيجة “مطابق/غير مطابق” دون score وreason.
- عدم حفظ نسخة القائمة المستخدمة.
- عدم وجود audit logs.
- استعمال live API وقت الفحص دون نسخة محلية أو fallback.
- عدم وجود حالة مراجعة للحالات عالية الخطورة.
- عدم وجود RBAC.

## 5. قرار الجاهزية

لا يُعتبر KYDEX جاهزاً للاستخدام المهني إلا إذا حقق 4/5 على الأقل في محاور: مصادر البيانات، المطابقة، سجل التدقيق، RBAC، ومسار التصعيد.
