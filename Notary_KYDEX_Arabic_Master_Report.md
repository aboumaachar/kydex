# الحزمة الموحّدة — بيئة كتاب العدل وKYDEX
# التقرير الجامع — بيئة كتاب العدل وKYDEX


> حزمة عمل عربية مبنية على خلاصة نقاشات بيئة كتاب العدل وKYDEX، وتشمل: محرك التحقق، موقع مجلس كتاب العدل، صفحة/ثيم الكاتب العدل، إضافات الربط API، تحليل عرض JAB، والطرح المقابل القائم على خدمات Web/SaaS بدلاً من تطبيقات الهاتف.

> ملاحظة منهجية: هذه الوثائق هي أدوات تدقيق واستراتيجية ومواصفات أولية. لا تُعد رأياً قانونياً نهائياً، ويجب مراجعتها من مستشار قانوني مختص قبل اعتمادها تعاقدياً أو تنظيمياً.


## 1. الغاية من الحزمة

تهدف هذه الحزمة إلى إقفال النقاش الاستراتيجي حول بيئة كتاب العدل الرقمية عبر تحويل الأفكار المتفرقة إلى منظومة متكاملة قابلة للتقديم والتدقيق والتنفيذ. المطلوب ليس بناء تطبيق هاتف، بل بناء خدمة مهنية رقمية مستمرة تخدم المجلس والكتاب العدل، وتؤسس في الوقت نفسه لمحرك KYDEX كمنتج SaaS قابل للتوسع.

## 2. المكونات الخمسة للبيئة المقترحة

### 2.1 محرك KYDEX للفحص والتحقق

KYDEX ليس “محرك بحث” بالمعنى التقليدي، بل طبقة استعلام وفحص متعددة المصادر. المستخدم يرسل استعلاماً واحداً، فيقوم النظام بتوحيد الاسم والبيانات، ثم يفحصها مقابل مصادر متعددة مثل OFAC، الأمم المتحدة، قوائم محلية، أو قوائم خاصة يوافق عليها صاحب النظام.

المخرجات المطلوبة:

- درجة تطابق.
- مصدر النتيجة.
- سبب التطابق.
- تاريخ إصدار/نسخة القائمة.
- مستوى المخاطر: منخفض، متوسط، عالٍ، حرج.
- سجل تدقيق قابل للإثبات.

### 2.2 صفحة مجلس كتاب العدل

المجلس يحتاج صفحة Web رسمية لا تكرر وظيفة التطبيق، بل تركز على:

- التعريف بالمجلس.
- نشر التعاميم والأخبار.
- دليل كتاب العدل.
- البحث عن الكاتب العدل حسب المنطقة.
- بوابة دخول الأعضاء.
- الربط بمحرك KYDEX عند الحاجة.
- دعم اسم النطاق الرسمي أو شبه الرسمي وفق ما يقرره المجلس.

### 2.3 موقع/ثيم خاص لكل كاتب عدل

كل كاتب عدل يمكن أن يملك صفحة مستقلة أو موقعاً صغيراً جاهزاً للنشر، يحتوي على:

- الاسم والصفة والصورة المهنية.
- العنوان وساعات العمل.
- الخريطة ومعلومات التواصل.
- نموذج طلب موعد.
- روابط المستندات المطلوبة.
- زر دخول أو ربط اختياري بخدمة KYDEX.

### 2.4 إضافات الربط API / WordPress / منصات خارجية

جوهر التوسع هو أن KYDEX لا يُحبس داخل واجهة واحدة. يجب أن يكون قابلاً للربط عبر:

- WordPress plugin.
- بوابات المجلس.
- مواقع الكتاب العدل.
- تطبيقات خارجية مرخصة.
- أنظمة داخلية للمؤسسات.

النموذج هو many-to-many: عدة مستخدمين وعدة تطبيقات يمكنهم الاستعلام عبر محرك موحد واحد، بينما يتولى KYDEX الوصول إلى عدة مصادر بيانات.

### 2.5 التشغيل اليومي والتدريب والصيانة

نجاح المشروع لا يعتمد على التطوير وحده. يجب تقديمه كخدمة تشغيل مستمرة تشمل:

- التطوير والتهيئة.
- التدريب وبناء القدرات.
- الصيانة والدعم.
- التحديث اليومي للقوائم والبيانات.
- إدارة المستخدمين.
- مراقبة الامتثال.
- دعم التصعيد عند وجود تطابق أو شك.

## 3. القرار الاستراتيجي: Web/SaaS لا تطبيق هاتف

عرض JAB يذهب باتجاه iOS وAndroid مع Web Admin. المقاربة المقابلة تقول إن الكاتب العدل يعمل من المكتب، أمام شاشة حاسوب، وفي سياق مستندات وسندات وملفات. لذلك فإن Web Platform أو PWA يكفي ويغطي عملياً وظائف الهاتف مع كلفة أقل وصيانة أبسط وتحكم أفضل بالبيانات والأمن.

## 4. نقاط القوة التي يجب أخذها من JAB

لا يجب رفض عرض JAB بالكامل. فيه عناصر مفيدة يجب إعادة توظيفها:

- فكرة دمج مصادر UN / OFAC / ISF أو ما يماثلها.
- استخدام Soundex أو البحث الصوتي/التقريبي.
- وجود حساب لكل مستخدم.
- وجود Web Admin.
- دليل كتاب العدل والخريطة.

لكن هذه العناصر يجب أن تصبح وحدات داخل نظام أكبر، لا أن تبقى تطبيق هاتف منفصل.

## 5. نقاط الضعف الجوهرية في JAB

- يحصر المشروع في تطبيق هاتف رغم أن الاستخدام مكتبي.
- يعرض وظيفة بحث، لا منظومة امتثال.
- لا يثبت العناية الواجبة ولا يحفظ نسخة القائمة المستخدمة عند الفحص.
- لا يقدم نظام تصعيد للحالات المشكوك فيها.
- لا يتناول مسؤولية تحديث البيانات وتشغيلها اليومي بشكل تعاقدي واضح.
- لا يشرح أمن البيانات ولا العزل بين المستخدمين.
- لا يقدم API قابل للتوسع كمنتج.
- لا يقدّم صفحات فردية للكتاب العدل.
- لا يتضمن نموذج تدريب وصيانة وتشغيل.

## 6. النموذج التجاري المقترح

بدل كلفة تأسيس ضخمة لتطبيق iOS/Android، يتم تقديم:

### 6.1 كلفة تأسيس أولية منخفضة

تشمل:

- صفحة المجلس.
- قالب الكاتب العدل الفردي.
- واجهة بحث/تحقق أولية.
- إعداد الحسابات والربط الأساسي.

### 6.2 اشتراك تشغيلي مستمر

يغطي:

- تحديث القوائم.
- الصيانة.
- الدعم.
- الاستضافة.
- أمن البيانات.
- التقارير.

### 6.3 مسؤولية الخدمة

المزود يتحمل مسؤولية:

- صحة عملية التحديث.
- حفظ نسخ القوائم.
- مراقبة الفشل.
- دعم المستخدمين.
- توثيق العمليات.

## 7. إطار التصعيد عند وجود تطابق أو شك

عند وجود HIGH أو CRITICAL match لا يجوز أن يكون النظام مجرد شاشة نتيجة. يجب أن ينشئ Case ويقفل الإجراء بانتظار المراجعة:

1. فحص من الكاتب العدل.
2. تصنيف النتيجة.
3. إنشاء ملف حالة إذا كان الخطر عالياً أو حرجاً.
4. مراجعة داخلية من مسؤول امتثال أو لجنة مراجعة في المجلس.
5. إعداد حزمة أدلة وتقرير قابل للإرسال عبر المسار الرسمي إذا استمر الاشتباه.

## 8. توصية ختامية

يجب تقديم المشروع إلى المجلس على أنه:

> خدمة رقمية Web/SaaS لإدارة الفحص والامتثال والتوثيق والربط، لا مجرد تطبيق هاتف للبحث في القوائم.

هذه الصياغة تضعك في موقع من يعيد تعريف المشروع، وتسمح لك بمواجهة عرض JAB دون الهجوم عليه مباشرة: عرضه يغطي جزءاً تقنياً محدوداً، أما المقاربة المقترحة فتغطي المنظومة المهنية والتشغيلية والقانونية كاملة.


# أداة تدقيق محرك KYDEX للفحص والتحقق


> حزمة عمل عربية مبنية على خلاصة نقاشات بيئة كتاب العدل وKYDEX، وتشمل: محرك التحقق، موقع مجلس كتاب العدل، صفحة/ثيم الكاتب العدل، إضافات الربط API، تحليل عرض JAB، والطرح المقابل القائم على خدمات Web/SaaS بدلاً من تطبيقات الهاتف.

> ملاحظة منهجية: هذه الوثائق هي أدوات تدقيق واستراتيجية ومواصفات أولية. لا تُعد رأياً قانونياً نهائياً، ويجب مراجعتها من مستشار قانوني مختص قبل اعتمادها تعاقدياً أو تنظيمياً.



## منهجية التقييم الموحدة

يعتمد التدقيق على سلم من 0 إلى 5:

| الدرجة | المعنى |
|---|---|
| 0 | غير موجود إطلاقاً |
| 1 | مذكور نظرياً فقط |
| 2 | موجود جزئياً دون اختبار أو توثيق |
| 3 | موجود ويعمل أساسياً |
| 4 | موجود ومختبر ومتكامل مع بقية النظام |
| 5 | موجود، مختبر، موثق، قابل للتدقيق، ومناسب للإطلاق |

قرار التدقيق:

- أقل من 2.5: غير قابل للاعتماد.
- من 2.5 إلى 3.5: يحتاج تحسينات جوهرية.
- من 3.5 إلى 4.5: قابل للتجربة المحدودة.
- من 4.5 إلى 5: قابل للإطلاق المنضبط.


## 1. نطاق التدقيق

تُستخدم هذه الأداة لتقييم جاهزية KYDEX بوصفه محرك فحص وتحقيق امتثال، وليس مجرد خانة بحث. يجب أن يتحقق المدقق من أن النظام قادر على استقبال طلب فحص، تطبيع البيانات، فحصها مقابل مصادر متعددة، عرض نتائج مفهومة، وحفظ دليل تدقيق كامل.

## 2. محاور التدقيق

### 2.1 مصادر البيانات والربط

| بند التدقيق | دليل الإثبات المطلوب | الدرجة |
|---|---|---|
| وجود Connector مستقل لكل مصدر بيانات | كود/وثيقة لكل OFAC/UN/محلي | 0-5 |
| عدم الاعتماد على استعلام live فقط وقت الفحص | قاعدة محلية محدثة ونسخ قوائم | 0-5 |
| حفظ نسخة القائمة المستخدمة في كل فحص | list_version في سجل الفحص | 0-5 |
| وجود آلية إنذار عند فشل التحديث | logs + alerting | 0-5 |
| دعم رفع قوائم يدوية CSV/JSON | upload + validation report | 0-5 |

### 2.2 محرك المطابقة

| بند التدقيق | دليل الإثبات المطلوب | الدرجة |
|---|---|---|
| Exact match | أمثلة اختبار | 0-5 |
| Fuzzy matching | نتائج وعتبات واضحة | 0-5 |
| Soundex أو بديل صوتي | مستند خوارزمية | 0-5 |
| تطبيع الأسماء العربية واللاتينية | transliteration rules | 0-5 |
| Alias matching | حقول aliases وربطها | 0-5 |
| شرح سبب النتيجة | match_reason في API/UI | 0-5 |

### 2.3 تصنيف المخاطر

| مستوى الخطر | معيار مقترح | الإجراء |
|---|---|---|
| LOW | لا تطابق أو تشابه ضعيف | متابعة + تسجيل |
| MEDIUM | تشابه جزئي | ملاحظة عناية واجبة |
| HIGH | تطابق قوي أو مؤشر هوية داعم | إنشاء حالة مراجعة |
| CRITICAL | تطابق شبه مؤكد | تجميد الإجراء وتحضير ملف تصعيد |

### 2.4 سجل التدقيق والأدلة

| بند التدقيق | دليل الإثبات المطلوب | الدرجة |
|---|---|---|
| تسجيل كل فحص | query log | 0-5 |
| تسجيل المستخدم والوقت والـ IP | audit metadata | 0-5 |
| تسجيل نسخة القوائم | source version | 0-5 |
| منع تعديل سجلات التدقيق | immutable / hash-chain | 0-5 |
| توليد evidence package | PDF/JSON package | 0-5 |
| تصدير timeline للحالة | case timeline export | 0-5 |

### 2.5 الأمن والخصوصية

| بند التدقيق | دليل الإثبات المطلوب | الدرجة |
|---|---|---|
| HTTPS/TLS | إعدادات الخادم | 0-5 |
| تشفير كلمات المرور | bcrypt/argon2 | 0-5 |
| API keys hashed | schema + tests | 0-5 |
| RBAC | Route protection matrix | 0-5 |
| 2FA للمسؤولين | شاشة/اختبار | 0-5 |
| عزل المستأجرين/العملاء | tenant_id enforcement | 0-5 |
| نسخ احتياطي مشفر | backup logs | 0-5 |

## 3. أسئلة التدقيق الحاسمة

1. هل يمكن إثبات أن الفحص تم على نسخة معينة من OFAC/UN؟
2. هل يستطيع النظام شرح سبب كل match؟
3. هل يعالج اختلافات كتابة الأسماء العربية والإنكليزية؟
4. هل يمنع المستخدم العادي من الوصول إلى ملفات الأدلة الحساسة؟
5. هل يوجد مسار مراجعة عند HIGH/CRITICAL؟
6. هل يمكن للمدقق إعادة بناء القرار بعد سنة أو خمس سنوات؟

## 4. مؤشرات الخطر الأحمر

- نتيجة “مطابق/غير مطابق” دون score وreason.
- عدم حفظ نسخة القائمة المستخدمة.
- عدم وجود audit logs.
- استعمال live API وقت الفحص دون نسخة محلية أو fallback.
- عدم وجود حالة مراجعة للحالات عالية الخطورة.
- عدم وجود RBAC.

## 5. قرار الجاهزية

لا يُعتبر KYDEX جاهزاً للاستخدام المهني إلا إذا حقق 4/5 على الأقل في محاور: مصادر البيانات، المطابقة، سجل التدقيق، RBAC، ومسار التصعيد.

# نقاط الضعف في عرض JAB — صياغة عربية للاستخدام مع المجلس


> حزمة عمل عربية مبنية على خلاصة نقاشات بيئة كتاب العدل وKYDEX، وتشمل: محرك التحقق، موقع مجلس كتاب العدل، صفحة/ثيم الكاتب العدل، إضافات الربط API، تحليل عرض JAB، والطرح المقابل القائم على خدمات Web/SaaS بدلاً من تطبيقات الهاتف.

> ملاحظة منهجية: هذه الوثائق هي أدوات تدقيق واستراتيجية ومواصفات أولية. لا تُعد رأياً قانونياً نهائياً، ويجب مراجعتها من مستشار قانوني مختص قبل اعتمادها تعاقدياً أو تنظيمياً.


## 1. خلاصة موجزة

عرض JAB يحتوي على فكرة تقنية مفيدة، وهي توفير بحث في قوائم عقوبات/قوائم سوداء من خلال تطبيق جوال ولوحة إدارة. لكنه لا يعالج المشروع كمنظومة امتثال وتشغيل مهنية، ولا يقدم نموذجاً مستداماً للتحديث اليومي، التدريب، الصيانة، المسؤولية، أو الربط الواسع عبر API.

الصياغة المقترحة أمام المجلس:

> العرض يغطي وظيفة تقنية محددة هي البحث في القوائم، لكنه لا يغطي الإطار التشغيلي والقانوني والمؤسساتي الكامل الذي يحتاجه مجلس كتاب العدل.

## 2. نقاط الضعف الجوهرية

### 2.1 اعتماد خاطئ على تطبيق iOS/Android

الكاتب العدل يباشر عمله في المكتب، ضمن ملفات وسندات وأجهزة حاسوب وطابعات. لذلك فإن تطبيق الهاتف ليس الوسيط الطبيعي للعمل اليومي. منصة Web تحقق الوظائف نفسها وتضيف سهولة الطباعة، حفظ الأدلة، ربط الملفات، والوصول من المكتب دون تكاليف متاجر التطبيقات.

### 2.2 تقليص المشروع إلى أداة بحث

عرض JAB يتعامل مع المشروع كـ blacklist lookup، بينما الحاجة الحقيقية هي منظومة امتثال تتضمن:

- فحص.
- عناية واجبة.
- توثيق.
- تصعيد.
- حفظ دليل.
- تقارير.
- إدارة مستخدمين.

### 2.3 غياب سجل تدقيق قانوني

لا يكفي أن تظهر نتيجة البحث. يجب أن يثبت النظام:

- من أجرى الفحص.
- متى.
- على أي نسخة من القوائم.
- ما النتيجة.
- ماذا فعل الكاتب العدل بعد النتيجة.

### 2.4 غياب إدارة الحالات المشكوك فيها

عند وجود تطابق أو شك، يجب ألا يترك الأمر لقرار شفهي أو فردي. يجب إنشاء case، تعيين مراجعة داخلية، تسجيل قرار، وتحضير ملف تصعيد إذا لزم الأمر.

### 2.5 ضعف تصور API

ذكر API وحده لا يكفي. المطلوب API قابل للتوسع، مفاتيح لكل مستخدم/تطبيق، صلاحيات، سجلات استخدام، versioning، rate limits، وإمكانية ربط منصات متعددة بمحرك موحد.

### 2.6 غياب مسؤولية التشغيل اليومي

أهم سؤال للمجلس ليس “من يطوّر التطبيق؟” بل:

- من يحدّث القوائم؟
- من يراقب فشل التحديث؟
- من يصحح البيانات؟
- من يدرب المستخدمين؟
- من يتحمل مسؤولية الاستمرارية؟

### 2.7 غياب الأمن والخصوصية

لا يظهر من العرض وجود مواصفات كافية لـ:

- تشفير.
- RBAC.
- 2FA.
- عزل المستخدمين.
- حماية المفاتيح.
- نسخ احتياطي.
- منع الوصول غير المرخص.

### 2.8 غياب صفحات المجلس والكتاب العدل

العرض يضع دليل كتاب العدل داخل التطبيق، لكنه لا يبني حضوراً رقمياً عاماً للمجلس ولا صفحات مستقلة للكتاب العدل. وهذا يضيّع فرصة تحويل المشروع إلى بيئة خدمات رقمية لا مجرد تطبيق مغلق.

### 2.9 غياب نموذج التدريب والتبني

أي نظام جديد سيفشل إذا لم يترافق مع:

- كتيبات.
- فيديوهات.
- تدريب حضوري/أونلاين.
- دعم استخدام.
- قياس adoption.

### 2.10 غياب النموذج التجاري المستدام

تطبيق الهاتف يخلق كلفة تأسيس وصيانة مستمرة، بينما نموذج Web/SaaS يسمح بتكلفة دخول أقل واشتراك مستمر يغطي التشغيل والتحديث والدعم.

## 3. نقاط القوة التي يمكن تبنيها

- تعدد المصادر.
- Soundex أو البحث التقريبي.
- حساب لكل كاتب عدل.
- Web Admin.
- دليل وخريطة.

لكن هذه النقاط يجب أن تدخل ضمن منصة Web/SaaS، لا أن تبقى تطبيقاً منفصلاً.

## 4. الخلاصة القابلة للاستخدام

لا ينبغي تقديم النقد على أنه رفض لشركة أو عرض، بل كإعادة تعريف للمشروع:

> المطلوب ليس تطبيق هاتف للبحث في القوائم، بل خدمة رقمية مؤسسية لإدارة الفحص والامتثال والتوثيق والتشغيل اليومي، ويمكن أن تتضمن كل وظائف عرض JAB ولكن من خلال Web Platform وAPI، بكلفة أقل واستدامة أكبر.

## 2. الملخص التنفيذي

يقترح هذا العرض اعتماد مقاربة Web/SaaS بدلاً من بناء تطبيقات iOS وAndroid. المقاربة الجديدة تحافظ على جميع الوظائف التي وردت في عرض JAB، لكنها تنقلها إلى بيئة أكثر ملاءمة لطبيعة عمل الكاتب العدل: صفحة المجلس، صفحة لكل كاتب عدل، بوابة دخول مكتبية، ومحرك KYDEX للفحص عبر API.

## 3. لماذا لا نحتاج iOS وAndroid؟

- العمل الفعلي للكاتب العدل يتم في المكتب وليس أثناء التنقل.

- Web Platform يعمل على الحاسوب والهاتف والتابلت دون متجر تطبيقات.
- أي تحديث في الويب يصل فوراً لكل المستخدمين دون تحميل نسخة جديدة.
- الصيانة والأمن والتحكم بالبيانات أسهل بكثير.

# لائحة التنفيذ والمتابعة — بيئة كتاب العدل وKYDEX


> حزمة عمل عربية مبنية على خلاصة نقاشات بيئة كتاب العدل وKYDEX، وتشمل: محرك التحقق، موقع مجلس كتاب العدل، صفحة/ثيم الكاتب العدل، إضافات الربط API، تحليل عرض JAB، والطرح المقابل القائم على خدمات Web/SaaS بدلاً من تطبيقات الهاتف.

> ملاحظة منهجية: هذه الوثائق هي أدوات تدقيق واستراتيجية ومواصفات أولية. لا تُعد رأياً قانونياً نهائياً، ويجب مراجعتها من مستشار قانوني مختص قبل اعتمادها تعاقدياً أو تنظيمياً.


## 1. قبل العرض على المجلس

- تجهيز ملخص صفحة واحدة: لماذا Web/SaaS أفضل من iOS/Android.
- تجهيز مقارنة مختصرة مع عرض JAB دون لهجة هجومية.
- تحديد ما سيبقى من عرض JAB: مصادر متعددة، Soundex، Web Admin، دليل وخريطة.
- تحديد ما يجب إضافته: audit، case management، training، maintenance، data operations.
- تحضير نموذج تكلفة تأسيس منخفض + اشتراك تشغيلي.
- تحضير نسخة demo أو mockup لصفحة المجلس والكاتب العدل.

## 2. أثناء العرض

الرسائل الأساسية:

1. التطبيق ليس الهدف؛ الخدمة هي الهدف.
2. الكاتب العدل يعمل من المكتب، لذلك Web Platform أكثر ملاءمة.
3. تحديث البيانات اليومي أهم من شكل الواجهة.
4. المسؤولية التشغيلية يجب أن تكون واضحة.
5. API يسمح بالربط مع عدة منصات الآن ومستقبلاً.
6. كل وظيفة في عرض JAB موجودة، لكن بطريقة أكثر استدامة.

## 3. بعد العرض

- طلب نسخة رسمية من عرض JAB وجداول الكلفة.
- طلب تحديد مسؤولية تحديث القوائم في عرض JAB.
- طلب security specification من JAB.
- طلب API specification من JAB.
- طلب SLA والصيانة والتدريب.
- مقارنة العرضين على أساس كلفة 3 سنوات لا كلفة التطوير فقط.

## 4. أسئلة يجب طرحها على أي مزود

1. هل النظام يحفظ نسخة القائمة المستخدمة في كل فحص؟
2. هل يوجد سجل تدقيق لا يمكن تعديله؟
3. من يحدّث البيانات يومياً؟
4. ماذا يحدث إذا فشل التحديث؟
5. هل يوجد مسار تصعيد عند HIGH/CRITICAL؟
6. هل يوجد API key لكل تطبيق؟
7. هل يمكن إيقاف مستخدم أو مفتاح API فوراً؟
8. هل يتم تدريب الكتاب العدل؟
9. ما كلفة الصيانة السنوية؟
10. هل يمكن للكاتب العدل استخدام النظام من الحاسوب والطابعة؟
